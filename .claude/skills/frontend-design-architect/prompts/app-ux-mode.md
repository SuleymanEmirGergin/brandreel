# App UX Mode

This mode is used when the user wants UI direction for **product interfaces** such as dashboards, admin panels, analytics tools, or productivity applications.

The goal is to optimize **clarity, scanning speed, and task completion**.

Unlike marketing pages, product interfaces prioritize **efficiency over persuasion**.

---

# When To Use

Use this mode when the user asks for:

- SaaS dashboard UI
- admin panel design
- analytics interface
- productivity tools
- internal tools
- data-heavy applications

Example requests:

- “Bir SaaS dashboard tasarla.”
- “Admin panel UI öner.”
- “Analytics tool için arayüz tasarla.”
- “Proje yönetim dashboard’u yap.”

---

# Core Philosophy

Product interfaces should help users:

- understand information quickly
- complete tasks efficiently
- navigate features easily

The UI should minimize friction and support **rapid decision making**.

---

# Key Principles

## 1. Information Hierarchy

Data should be easy to scan.

Use:

- clear section titles
- grouped information
- visual emphasis for key metrics

Avoid:

- flat layouts
- equal visual weight everywhere

---

## 2. Navigation Clarity

Users must understand where they are and where they can go.

Common navigation patterns:

- sidebar navigation
- top navigation
- tab navigation
- breadcrumb navigation

Navigation should be predictable and consistent.

---

## 3. Data Density

Dashboards often display large amounts of data.

Balance:

- information density
- readability

Use:

- stat cards
- tables
- charts
- filters

Avoid overwhelming the user with too much information at once.

---

## 4. Fast Interaction

Product UI should support quick actions.

Important elements:

- search
- filters
- quick actions
- keyboard-friendly inputs

Users should complete tasks with minimal steps.

---

## 5. Component Reusability

Product UI should rely on a strong component system.

Common components:

- navigation sidebar
- top bar
- stat cards
- data tables
- filters
- modals
- forms
- notifications
- empty states

Reusable components improve consistency and development speed.

---

# Layout Patterns

Typical dashboard layouts include:

Sidebar navigation

- Top navigation
- Main content area

Main content may include:

- stat overview
- charts
- data tables
- activity feed
- detail panels

Layouts should prioritize **scanability**.

---

# Data Visualization

When data is present, use:

- charts
- graphs
- progress indicators
- visual summaries

Charts should emphasize insights rather than decoration.

Avoid unnecessary complexity.

---

# Interaction Patterns

Common interactions include:

- filtering
- sorting
- searching
- pagination
- inline editing

Interfaces should support **efficient workflows**.

---

# Screen Structure

Product interfaces often include screens such as:

- dashboard overview
- item list
- item detail
- analytics view
- settings
- profile
- activity logs

Each screen should have a clear purpose.

---

# Output Structure

When using App UX Mode, structure the response like this.

## 1. Product Context

Explain the product type and user workflow.

---

## 2. Information Architecture

Describe the structure of the application.

Example:

- dashboard
- projects
- analytics
- settings

---

## 3. Screen Structure

Explain the layout of major screens.

Example:

- overview dashboard
- data tables
- detail views

---

## 4. Component System

Examples:

- sidebar
- top navigation
- cards
- tables
- filters
- modals

---

## 5. Interaction Patterns

Explain how users interact with the UI.

Examples:

- filtering
- sorting
- search
- inline actions

---

## 6. Responsive Behavior

Explain how the dashboard adapts to smaller screens.

Example:

- collapsed sidebar
- stacked cards
- simplified tables

---

# Anti-Patterns

Avoid:

- cluttered dashboards
- unclear navigation
- excessive visual noise
- unnecessary animations
- dense unreadable tables

Product UI must prioritize **clarity and speed**.

---

# Success Criteria

A strong App UX Mode answer should:

- create a clear information hierarchy
- support fast workflows
- organize complex data clearly
- rely on reusable components
- remain scalable as the product grows
